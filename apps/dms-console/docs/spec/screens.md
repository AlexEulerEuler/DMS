# screens.md — 라우트별 화면 개발 스펙

> 범위: `apps/dms-console` 콘솔 한정. 이 문서는 [README.md](./README.md) 7절의 모든 라우트를 화면 단위로 구체화한다. enum·색·리스트 규약은 README를 정본으로 삼고, 컴포넌트명은 [components.md](./components.md), 호출 API는 [api-contract.md](./api-contract.md), 타입은 [data-schema.md](./data-schema.md)와 일치시킨다(세 문서는 병행 작성 중이며 이름·경로 관례를 README/IA에 맞춘다).

## 0. 공통 규칙 (모든 화면 적용)

이 절은 각 화면에서 반복하지 않는 공통 사항이다.

- **앱 셸**: 모든 화면은 `AppShell`(상단 바 `TopNav` + 좌측 `SideNav` + 본문) 안에서 본문만 교체된다. 셸은 상태 전이 중에도 항상 렌더된다([foundation.md](../ia/foundation.md) 2절).
- **공통 상태 4종**: 빈·로딩·에러·없음은 `StateShell`을 재사용하고 문구만 맥락에 맞춘다. 아래 표는 각 화면의 "없음" 조건만 개별 명시한다.
  - 로딩: 콘텐츠 자리에 스켈레톤. 셸 유지.
  - 빈: 로드 성공·0건. 안내 + 주 액션 진입점.
  - 에러: 로드/저장 실패. 메시지 + "다시 시도". 셸·내비게이션 유지.
  - 없음(not-found): 정의되지 않은 경로·존재하지 않는 식별자. "없음" 화면 표시, **자동 리다이렉트 없음**. (모듈 루트→기본 화면 리다이렉트만 예외.)
- **상태·우선순위 표기**: `StatusBadge`/`PriorityBadge`는 README 4절 색을 쓰되 **항상 텍스트 라벨과 함께** 노출(색만으로 의미 구분 금지). 코드에서는 enum key, 화면에서는 한국어 라벨.
- **리스트 공통 규약**(README 5절): 페이지네이션 페이지 크기 25, 정렬 기본 등록일 최신순, 상태 필터 기본, 제목 검색, 총 개수 헤더 요약. Issues만 우선순위 정렬·필터 추가.
- **enum 참조**(README 3절): `CommonStatus`(진행예정/진행중/완료), `WorkStatus`(+리뷰중), `AgentStatus`(기획중/확정/보류), `IssueState`(open/closed), `Priority`(긴급/높음/보통/낮음, 기본 normal), `TaskNodeType`(구분/중분류/세부 업무).
- **인증·연동**(README 6절): 콘솔 자체 API는 단일 사용자·전체 접근. GitHub는 읽기 전용 미러 + 새 이슈 생성(write-through), 진입 시 조회 + 수동 새로고침.
- **접근성**: 포커스 순서 상단→좌측→본문, 키보드 도달 가능, 상태를 색+텍스트로 전달([foundation.md](../ia/foundation.md) 10절).

라우트별 상세는 아래. 순서는 [ia.md](../ia.md) 10절 MVP 우선순위를 따른다.

# A. MVP 화면 (ia.md 10절 우선순위)

## A-1. `/overview` — 프로젝트 개요

- **화면명**: 프로젝트 개요 (Overview 기본 화면, 앱 루트 `/`는 여기로 리다이렉트)
- **사용 컴포넌트**: `PageHeader`(제목 "프로젝트 개요" + 한 줄 설명), `MetaChipRow`(버전·언어·브랜치·패키지·API 읽기 전용 배지), 개요 문단, `CardGrid` + `SummaryCard`(입력 카드 3 + 출력 카드 1), `WorkflowDiagram`(단계 흐름, 데이터 기반 렌더), `SideNav`(프로젝트 문서 12종)
- **데이터(호출 API)**: `GET /api/overview/meta` → `ProjectMeta`(version·language·branch·package·apiEndpoint), `GET /api/overview/pipeline-summary` → `PipelineSummary`(inputs[]·outputs[]·workflowSteps[]). 원격 API 런타임 조회.
- **상태**: 빈=요약/메타 미제공 시 해당 카드 영역만 비우고 안내 문구 / 로딩=헤더·카드 스켈레톤 / 에러=메시지 + 다시 시도(사이드바는 계속 동작) / 없음=해당 없음(기본 라우트)
- **주요 액션**: 좌측 사이드바로 12개 문서 전환(선택 항목 파란 배경), 카드·워크플로우 내 참조에서 관련 모듈로 이동(예: 일정 생성 → WBS). 메타 칩은 읽기 전용(클릭 없음). 생성·편집·삭제 없음.
- **수용 기준**:
  - [ ] `/`로 진입하면 `/overview`로 리다이렉트되고 프로젝트 개요가 렌더된다.
  - [ ] 메타 로드가 성공하면 버전·언어·브랜치·패키지·API 칩이 값과 함께 표시된다.
  - [ ] 메타 값이 일부 누락되면 값 없는 칩은 숨기고 나머지는 정상 표시된다.
  - [ ] 입력 3종·출력 1종 카드와 워크플로우가 `PipelineSummary` 데이터로 렌더된다.
  - [ ] 로드가 실패하면 "다시 시도"가 노출되고, 눌러 재시도하면 로딩 상태로 돌아간다.
  - [ ] 좌측에서 다른 문서를 선택하면 콘텐츠만 교체되고 상단 바·사이드바는 유지된다.

## A-2. `/task/ia` — 업무 IA 트리

- **화면명**: IA (업무 정보구조 트리, Task 기본 화면. `/task`는 여기로 리다이렉트)
- **사용 컴포넌트**: `PageHeader`(제목 "IA" + 설명), `StatusFilterTabs`(전체/완료/진행중/진행예정 단일 선택), `AddItemButton`(항목 추가), `TaskTree`(구분 노드=펼침·접힘 토글·라벨·View 액션 / 세부 업무 행=라벨·`StatusBadge`·담당자·기한), `TaskNodeForm`(추가·편집: 라벨·상위 노드·상태·담당자·기한 시작·종료), `EmptyState`
- **데이터(호출 API)**: `GET /api/tasks` → `TaskIA[]`(트리, `type`=category|group|task). 등록/수정 `POST /api/tasks`·`PATCH /api/tasks/:id`, 삭제 `DELETE /api/tasks/:id`. 세부 업무 저장 시 대응 `WBSItem`이 서버에서 1:1 자동 파생(`linkedWbsId`).
- **상태**: 빈=등록 업무 없음 안내 + 항목 추가 유도 / 로딩=트리 스켈레톤 / 에러=메시지 + 다시 시도 / 없음=경로 자체는 고정(리다이렉트 대상). 필터 결과 0건은 트리 골격 유지 + "해당 상태의 업무가 없습니다"
- **주요 액션**: 항목 추가(구분/중분류/세부 업무 — 세부 업무는 상위 노드·초기 상태 지정), View(노드 상세·요약), 상태 필터, 구분 단위 펼침/접힘, 세부 업무 상태·일정 변경(연결 WBS 막대·진행률 자동 반영). 구분·중분류 삭제는 확인 후 하위 연쇄 삭제.
- **수용 기준**:
  - [ ] 트리가 로드되면 구분 노드 아래 세부 업무가 들여쓰기로 표시되고 각 행에 상태·담당자·기한이 나온다.
  - [ ] 상태 필터에서 "진행중"을 고르면 진행중 세부 업무만 남고, 결과가 없으면 "해당 상태의 업무가 없습니다"가 골격과 함께 표시된다.
  - [ ] 세부 업무를 추가·저장하면 트리에 반영되고 대응 WBS 항목이 1:1로 파생된다.
  - [ ] 구분 노드의 토글을 누르면 하위 항목이 펼쳐지거나 접힌다.
  - [ ] 구분/중분류를 삭제하면 확인 후 하위 노드가 연쇄 삭제된다.
  - [ ] 계층은 구분–중분류–세부 업무 3단을 넘기는 입력이 허용되지 않는다.

## A-3. `/wbs` — 진행현황 (WBS)

- **화면명**: 진행현황 (WBS, WBS 기본·단일 화면)
- **사용 컴포넌트**: `PageHeader`(제목·과제명·설명), `Legend`(완료/진행 중/진행 예정 색 키), `OverallProgress`(수치 + 막대), `GanttGrid`(좌측 정보 열=구분·세부 업무·진행상태 sticky / 시간축 헤더=월·주차 2단 / `GanttBar` 상태별 색 / 그리드 셀·오늘 강조선), `MilestoneMarker`, 그룹 접기/펼치기, `EmptyState`
- **데이터(호출 API)**: `GET /api/wbs` → `WBSItem[]`(Task 세부 업무 type=task에서 1:1 파생 투영: group·title·status·owner·startDate·endDate·progress·order). 시간축(월/주차)은 저장값이 아니라 최소 startDate~최대 endDate에서 파생 계산. 조회 전용(편집 없음).
- **상태**: 빈=업무 0건 안내 + 전체 진행률 표시 보류 또는 0% / 로딩=그리드 스켈레톤 / 에러=메시지 + 다시 시도 / 없음=단일 라우트(해당 없음)
- **주요 액션**: 가로 스크롤(시간축 헤더·좌측 열 고정), 세로 스크롤, 막대·행 호버 툴팁(시작~종료일·상태·진행률), 진입 시 오늘 주차로 자동 스크롤(기본 on), 그룹 접기/펼치기, 마일스톤 확인. **막대 직접 편집 없음**(일정 변경은 Task 흐름).
- **수용 기준**:
  - [ ] 진입하면 좌측 열(구분·세부 업무·진행상태)과 월/주차 시간축 그리드가 렌더되고 현재 주차가 보이도록 자동 스크롤된다.
  - [ ] 상태가 완료면 파란 막대, 진행중이면 초록 막대, 진행예정이면 회색/빈 일정으로 표시된다.
  - [ ] 전체 진행률은 업무별 진행률 단순 평균으로 계산되어 수치와 막대로 표시된다(progress 없으면 완료=100·진행중=50·진행예정=0으로 환산).
  - [ ] 가로로 스크롤해도 좌측 정보 열과 시간축 헤더가 고정된다.
  - [ ] 시작일/종료일이 없는 업무는 막대 없이 좌측 열에만 표시된다.
  - [ ] 종료일이 시작일보다 빠른 데이터는 경고로 표시되고 막대는 시작 주차만 그린다.

## A-4. `/issues` — 이슈 목록

- **화면명**: Issues (이슈 목록, Issues 기본 화면 = 모듈 루트, 리다이렉트 없음)
- **사용 컴포넌트**: `PageHeader`(제목·설명·요약 "열린 이슈 N"), `NewIssueButton`(→ `/issues/new`), `DataTable`(열: #·제목·상태·우선순위·라벨·담당자, 행 클릭 시 상세), `StatusBadge`(open/closed)·`PriorityBadge`·`LabelChip`, `ArchiveFilterToggle`(닫힌 이슈), `RefreshButton`(수동 새로고침), `StatusFilter`, `SearchInput`, `Pagination`, `EmptyState`
- **데이터(호출 API)**: `GET /api/issues?state=open|closed&priority=&q=&page=` → `Issue[]`(GitHub 미러 + 로컬 `priority`·`linkedWorkItems`). 진입 시 GitHub 조회 + 수동 새로고침.
- **상태**: 빈=이슈 0건 안내 + 등록 유도 / 로딩=테이블 스켈레톤 / 에러=GitHub 연동 실패 메시지 + 다시 시도 / 없음=모듈 루트라 해당 없음. 닫힌 이슈는 기본 숨김(아카이브 필터로 조회)
- **주요 액션**: 이슈 등록 진입, 행 클릭으로 상세, 아카이브 필터로 닫힌 이슈 조회, 우선순위·상태 필터, 제목 검색, 수동 새로고침. 정렬은 우선순위(긴급→낮음) → 등록일 최신순.
- **수용 기준**:
  - [ ] 진입하면 열린 이슈가 우선순위 → 등록일 순으로 정렬되어 테이블에 표시되고 헤더에 열린 이슈 수가 나온다.
  - [ ] 아카이브 필터를 켜면 닫힌(closed) 이슈가 조회된다.
  - [ ] 각 행의 상태·우선순위·라벨이 배지/칩으로 텍스트 라벨과 함께 표시된다.
  - [ ] 우선순위가 지정되지 않은 이슈는 기본값 "보통"으로 표시된다.
  - [ ] 행을 클릭하면 `/issues/:issueId` 상세로 이동한다.
  - [ ] GitHub 연동이 실패하면 에러 안내와 "다시 시도"가 노출된다.

## A-5. `/work/backlog` — 작업 백로그

- **화면명**: Backlog (작업 백로그, Work 기본 화면. `/work`는 여기로 리다이렉트)
- **사용 컴포넌트**: `PageHeader`(제목 "Backlog" + 설명 + 요약 "총 작업 수"), `NewWorkButton`(→ `/work/new`), `StatusFilter`(진행예정/진행중/리뷰중/완료), `SearchInput`, `DataTable`(열: 작업명·담당·진행상태·진행 일정(시작~종료), 행에서 View로 상세), `StatusBadge`, `Pagination`, `EmptyState`
- **데이터(호출 API)**: `GET /api/work?status=&q=&page=` → `WorkItem[]`(title·owner·status·startDate·endDate·description·linkedIssue·linkedAgent). Backlog·Kanban은 동일 집합 공유.
- **상태**: 빈=작업 0건 안내 + 작업 등록 유도 / 로딩=테이블 스켈레톤 / 에러=메시지 + 다시 시도 / 없음=기본 라우트(해당 없음). 담당·일정 미지정 항목은 목록에서 "미지정"으로 구분 표시
- **주요 액션**: 작업 등록 진입, View로 상세, 상태 필터, 제목 검색. 정렬 기본 등록일 최신순.
- **수용 기준**:
  - [ ] `/work`로 진입하면 `/work/backlog`로 리다이렉트되고 작업 테이블과 총 작업 수가 표시된다.
  - [ ] 상태 필터를 고르면 해당 상태의 작업만 남는다.
  - [ ] 담당자나 일정이 없는 작업은 "미지정"으로 구분되어 표시된다.
  - [ ] 행의 View를 누르면 `/work/:workId` 상세로 이동한다.
  - [ ] 작업 등록 버튼을 누르면 `/work/new`로 이동하고, 등록 완료 후 직전 뷰(Backlog)로 복귀한다.
  - [ ] 작업이 0건이면 빈 상태 안내와 작업 등록 진입점이 표시된다.

## A-6. `/agents` — 에이전트 목록

- **주요 액션**: 사이드바로 문서 전환, 문서 내 참조 링크로 관련 모듈 이동. "주요 데이터"에서만 산출물 조회·파일 다운로드(생성 실행은 CLI/백엔드).
- **수용 기준**:
  - [ ] 정의된 slug(예: `/overview/glossary`)로 진입하면 해당 문서 제목·설명·콘텐츠가 렌더된다.
  - [ ] 정의되지 않은 slug(예: `/overview/unknown`)로 진입하면 "없음" 화면이 표시되고 리다이렉트되지 않는다.
  - [ ] 문서 콘텐츠 로드가 실패하면 "다시 시도"가 노출되고 사이드바 내비게이션은 계속 동작한다.
  - [ ] `/overview/data`에서는 표준 목록·생성 일정·내보내기 파일이 조회되고 파일을 다운로드할 수 있다.
  - [ ] 표시할 내용이 없는 문서는 "문서 준비 중" 안내와 다른 문서 링크를 보여 준다.

## C. "없음"(not-found) 처리 요약

정의되지 않은 경로·존재하지 않는 식별자는 항상 "없음" 화면을 표시하고 **자동 리다이렉트하지 않는다**(README 7절 · [foundation.md](../ia/foundation.md) 7~8절). 모듈 루트→기본 화면 리다이렉트만 정상 동작으로 예외.

| 상황 | 처리 |
| --- | --- |
| 정의되지 않은 경로(예: `/foo`, `/overview/unknown`) | "없음" 화면 |
| 존재하지 않는 `:issueId` | "없음" 화면 |
| 존재하지 않는 `:agentId` | "없음" 화면 |
| 유효하지 않은 `:workId` | 오류 안내 + 목록 복귀 경로([work.md](../ia/work.md) 10절 규정) |
| `/` · `/task` · `/work` | 기본 화면으로 리다이렉트(정상 동작, 예외) |
| `/issues` · `/agents` | 모듈 루트가 곧 목록(리다이렉트 없음) |
