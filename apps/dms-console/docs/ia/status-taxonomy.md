# Status Taxonomy — 공통 상태 체계

## 1. 개요

이 문서는 DMS Console 전반에서 쓰는 상태(Status) 값의 단일 기준이다. 공통 상태와 모듈별(Agent, Issue, Work) 상태를 한곳에 정의하고, 각 값의 의미, 색상·시각 표기, 전이 규칙, Task 상태 필터, 모듈 간 매핑을 정리한다.

- 목적: 모듈마다 제각기 정의될 수 있는 상태 명칭·색상·전이를 하나로 모아 일관성을 확보한다.
- 범위: 상태 값과 우선순위의 정의·규칙까지. 각 화면에서 상태가 어떻게 배치·조작되는지는 해당 페이지 문서로 링크한다(10절).
- 우선순위: 다른 페이지 문서가 상태 명칭·색상·전이를 언급할 때는 이 문서를 정본으로 삼는다. 화면 문서는 이를 표현·조작하는 계층이다.

용어 정리:
- 상태(Status): 항목이 지금 어느 단계에 있는지 나타내는 값. 한 항목은 한 시점에 하나의 상태만 가진다.
- 종료 상태(Terminal): 정상 흐름의 끝. 재작업·재개 같은 예외가 아니면 더 나아가지 않는다.
- 전이(Transition): 한 상태에서 다른 상태로의 이동. 허용된 전이만 UI에서 노출한다.

## 2. 공통 상태

계획·진행 성격의 항목(Task 세부 업무, WBS 업무 행)이 공유하는 기본 3단계다.

| 값 | 표기(공백형) | 의미 |
| --- | --- | --- |
| 진행예정 | 진행 예정 | 등록만 되고 아직 착수 전 |
| 진행중 | 진행 중 | 착수했고 완료 전 |
| 완료 | 완료 | 마무리된 종료 상태 |

- 저장·필터에서는 공백 없는 형태(`진행예정 / 진행중 / 완료`)를 값으로 쓰고, WBS 범례처럼 읽기 위주 화면에서는 공백형(진행 예정, 진행 중)을 라벨로 노출한다. 같은 값의 표기 차이일 뿐 별개 상태가 아니다.
- 모듈별 상태(3절)는 이 3단계를 세분하거나 이름을 달리한 것으로, 8절 매핑표에서 공통 상태로 환산한다.

## 3. 모듈별 상태

공통 3단계로는 부족한 모듈은 자체 상태 enum을 가진다. 값의 의미는 4절, 색상은 5절, 전이는 6절을 참조한다.

### 3.1 Agent 상태

에이전트 기획의 진행 단계를 나타낸다. 값은 아래와 같고, 전이 규칙은 6.2를 따른다.

```text
기획중 · 확정 · 보류
```

| 값 | 의미 |
| --- | --- |
| 기획중 | 기획 메모를 작성·수정 중. 등록 직후 기본값 |
| 확정 | 기획이 정리된 상태(종료) |
| 보류 | 잠시 멈춘 상태. 재개 시 기획중 복귀 |

> 실행·검증 관련 상태(테스트·배포 등)는 Agent 런타임 고도화 시 추가한다([agent.md](./agent.md)).

### 3.2 Issue 상태

이슈 상태는 GitHub의 open/closed를 그대로 따른다(GitHub 정본). 콘솔은 별도 상태 기계를 두지 않는다.

| 값 | 의미 |
| --- | --- |
| open | 열린 이슈. 목록 기본 노출 |
| closed | 닫힌(완료) 이슈. 아카이브로 조회 |

- 세분화된 진행 단계가 필요하면 GitHub 라벨로 표현하고, 콘솔은 이를 라벨로 표시한다(별도 상태 아님).
- 완료는 GitHub에서 닫힐 때(예: 연결 PR 병합) 자동 반영된다. 화면 동작은 [issues.md](./issues.md).

우선순위(priority)는 상태와 독립된 콘솔 로컬 축이며, 값 체계는 9절(우선순위)에서 정의한다.

### 3.3 Work 상태

개발 작업(백로그·Kanban 카드)의 진행 단계를 나타낸다.

```text
진행예정 → 진행중 → 리뷰중 → 완료
```

| 값 | Kanban 열 | 의미 |
| --- | --- | --- |
| 진행예정 | Todo | 할당됐으나 착수 전 |
| 진행중 | In Progress | 작업 진행 중 |
| 리뷰중 | Review | 결과 검토·리뷰 단계 |
| 완료 | Done | 작업 종료 상태 |

Work는 공통 3단계 사이에 리뷰중을 추가한 형태다. Kanban 보드의 열 이름(Todo/In Progress/Review/Done)은 같은 값의 영문 표기이며, 매핑은 8절에 정리한다.

## 4. 상태 값 정의·의미

세 모듈의 상태를 성격별로 묶으면 다음과 같다. 명칭이 달라도 역할이 같은 값끼리 이 문서에서 정렬해 둔다.

| 성격 | 공통 | Agent | Issue | Work |
| --- | --- | --- | --- | --- |
| 대기/시작 전 | 진행예정 | — | — | 진행예정 |
| 진행 | 진행중 | 기획중 | open | 진행중 · 리뷰중 |
| 종료 | 완료 | 확정 | closed | 완료 |
| 정지/보류 | — | 보류 | — | — |

- "종료"는 정상 흐름의 끝을 뜻한다. Agent 확정은 기획 완료 지점, Issue closed는 GitHub에서 닫힌 상태다.
- "정지/보류"는 흐름 바깥에서 진행을 멈춘 상태다. 재개 시 이전 단계로 되돌아간다(6절).
- Issue는 GitHub open/closed 2단계뿐이라 세분 칸이 없다. 세분 단계가 필요하면 GitHub 라벨을 쓴다.

## 5. 색상·시각 표기 규칙

상태는 색상과 형태로 구분한다. 색은 보조 신호이므로 항상 텍스트 라벨(배지/칩)을 함께 노출해 색만으로 의미가 갈리지 않게 한다.

### 5.1 공통 상태 색상(WBS 막대 포함)

WBS 일정표의 막대 색이 공통 상태 색상의 기준이다.

| 상태 | 색상 | WBS 막대 표현 |
| --- | --- | --- |
| 완료 | 파랑 | 파란 막대(채움) |
| 진행중 | 초록 | 초록 막대(채움) |
| 진행예정 | 회색 | 회색 막대 또는 빈 일정 |

```text
완료      ██████   파랑
진행중    ██████   초록
진행예정  ░░░░░░   회색 / 빈 일정
```

- 범례는 완료 / 진행 중 / 진행 예정 순으로 색 키를 보인다.
- 배지·칩·막대 어디에 쓰든 위 색-상태 대응은 동일하다.

### 5.2 모듈별 상태 색상 지침

모듈 상태는 공통 상태의 성격(4절)에 맞춰 색 계열을 정한다. 정확한 색상 값은 [../spec/design-tokens.md](../spec/design-tokens.md)를 따른다.

| 성격 | 색 계열 | 해당 값 |
| --- | --- | --- |
| 대기/시작 전 | 회색 | 진행예정 |
| 진행 | 초록 | 기획중, open, 진행중, 리뷰중 |
| 종료 | 파랑 | 확정, closed, 완료 |
| 정지/보류 | 앰버·주황 계열 | 보류(Agent) |

- 정지/보류는 흐름 이탈을 눈에 띄게 하려고 별도 경고 계열을 쓴다.
- 리뷰중은 진행 계열이되, 진행중과 구분이 필요하면 같은 계열의 다른 명도(예: 진행중과 톤 차이)로 표현한다.

## 6. 상태 전이 규칙

각 모듈의 허용 전이와 종료 상태를 정의한다. UI는 허용된 전이만 노출하고, 역전이는 재작업·재개 같은 명시적 상황에서만 허용한다.

### 6.1 공통 상태(Task · WBS)

```text
진행예정 ──▶ 진행중 ──▶ 완료
   ▲           │
   └───────────┘   (재작업 시 역전이 허용)
```

- 정상 흐름은 단방향(진행예정 → 진행중 → 완료).
- 완료 → 진행중 등의 역전이는 재작업 시 허용한다.
- 종료 상태: 완료.

### 6.2 Agent

```text
기획중 ──▶ 확정
  │
  └──▶ 보류 ──▶ (기획중 재개)
```

- 정상 흐름: 기획중 → 확정.
- 보류는 기획중에서 진입하고, 재개하면 기획중으로 돌아간다.
- 종료 상태: 확정. (실행·배포 관련 상태는 향후 Agent 런타임 고도화 시 추가)

### 6.3 Issue

```text
open ──(GitHub에서 닫힘: 예) 연결 PR 병합)──▶ closed
closed ──(재오픈)──▶ open
```

- 상태 전이의 정본은 GitHub다. 콘솔은 open/closed를 반영만 한다.
- 완료(closed)는 대개 연결 작업/PR이 GitHub 이슈를 닫으면서 일어난다.
- 종료 상태: closed(재오픈 가능).

### 6.4 Work

```text
진행예정 ──▶ 진행중 ──▶ 리뷰중 ──▶ 완료
   ▲          ▲          │
   └──────────┴──────────┘   (반려/재작업 시 역전이 허용)
```

- 정상 흐름: 진행예정 → 진행중 → 리뷰중 → 완료.
- 리뷰중에서 반려되면 진행중으로 되돌아갈 수 있다.
- 종료 상태: 완료.

### 6.5 종료 상태 요약

| 모듈 | 종료 상태 | 비고 |
| --- | --- | --- |
| 공통(Task·WBS) | 완료 | 재작업 시 역전이 가능 |
| Agent | 확정 | 보류는 일시 정지, 재개 시 기획중 |
| Issue | closed | GitHub 정본. 재오픈 가능 |
| Work | 완료 | 리뷰중에서 반려 시 역전이 |

## 7. Task 상태 필터

Task 화면의 상태 필터는 공통 상태에 "전체"를 더한 단일 선택이다.

| 필터 값 | 결과 |
| --- | --- |
| 전체 | 상태와 무관하게 모두 표시(기본값) |
| 완료 | 완료 세부 업무만 |
| 진행중 | 진행중 세부 업무만 |
| 진행예정 | 진행예정 세부 업무만 |

- 필터 값은 공통 상태(2절)와 동일한 저장형 명칭(공백 없음)을 쓴다.
- 필터 결과가 비면 트리 골격은 유지하고 "해당 상태의 업무가 없습니다" 안내를 노출한다(동작 상세는 `./task.md`).

## 8. 모듈별 상태 매핑표

모듈 상태를 공통 상태로 환산하는 기준이다. 대시보드 집계나 진행률 산출처럼 여러 모듈을 한 축으로 볼 때 이 매핑을 쓴다.

| 공통 상태 | Agent | Issue | Work | Work·Kanban 열 |
| --- | --- | --- | --- | --- |
| 진행예정 | — | — | 진행예정 | Todo |
| 진행중 | 기획중 | open | 진행중 · 리뷰중 | In Progress · Review |
| 완료 | 확정 | closed | 완료 | Done |
| (매핑 외) | 보류 | — | — | — |

- Agent 보류는 정상 3단계에 대응하지 않는 정지 상태다. 집계에서는 별도 범주로 두거나 제외한다.
- Work의 리뷰중은 공통상 "진행중"으로 환산한다. 완료로 보지 않는다.
- 진행률 환산이 필요하면 완료=100%, 진행중=50%, 진행예정=0%를 기준으로 하되, 세부 산출은 `./wbs.md`를 따른다.

## 9. 우선순위(Priority)

이슈처럼 우선순위 축을 갖는 항목에 적용한다. 상태(Status)와 독립된 축이며, 처리 순서·긴급도를 나타낸다. 우선순위 값의 단일 기준은 이 절이다.

| 값 | 의미 | 색 계열 |
| --- | --- | --- |
| 긴급 | 즉시 대응이 필요 | 적색 |
| 높음 | 우선 처리 대상 | 주황 |
| 보통 | 기본값 | 중립(회색) |
| 낮음 | 후순위 | 옅은 회색 |

- 기본값은 보통. 저장형 명칭은 공백 없이 `긴급 / 높음 / 보통 / 낮음`.
- 상태와 독립 축이므로 어떤 상태에서도 우선순위는 유지된다.
- 목록 기본 정렬은 우선순위 긴급 → 낮음, 동순위는 등록일 최신순(리스트 공통 규약은 [foundation.md](./foundation.md) 참조).
- 색상 값은 [../spec/design-tokens.md](../spec/design-tokens.md)를 따른다. 색만으로 구분하지 않고 라벨을 함께 노출한다.
