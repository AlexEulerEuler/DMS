# WBS — 진행현황

## 1. 개요

WBS 페이지는 대상 과제(프로젝트별 지정)의 세부 업무를 월/주 단위로 배치하고, 각 업무의 진행 상태를 간트형 일정표로 시각화하는 화면이다. 담당자와 이해관계자가 "지금 어디까지 왔고, 무엇이 지연되고 있으며, 다음에 무엇이 시작되는가"를 한눈에 파악하도록 돕는다.

Task 페이지가 개별 업무의 목록·속성을 다룬다면, WBS는 동일한 업무들을 시간축 위에 펼쳐 일정과 진행률의 관점으로 재구성한다. 데이터 자체는 공유하되 표현 축(시간)과 집계(진행률)가 다르다.

- 목적: 과제 전체 일정의 진행 현황을 시간축 기준으로 조망
- 핵심 산출물: 월별·주차별 간트 그리드, 전체 진행률
- 읽기 중심 화면(현재 범위에서는 편집보다 조회·모니터링 용도)

## 2. 내비게이션 위치

1차 메뉴 6개(Overview, Task, WBS, Issues, Work, Agent) 중 세 번째 항목이다.

```text
DMS Console
├── Overview
├── Task
├── WBS            ← 현재 문서
│   └── 진행현황    (/wbs)
├── Issues
├── Work
└── Agent
```

현재 WBS 하위에는 진행현황 단일 화면만 존재한다. 메뉴 항목 클릭 시 곧바로 `/wbs`로 진입한다(중간 랜딩 없음).

## 3. 하위 구조 / 라우트

| 경로 | 화면 | 설명 |
| --- | --- | --- |
| `/wbs` | 진행현황(WBS) | 월/주 단위 간트형 일정·진행 상태 표시 |

- 하위 라우트·상세 진입 경로는 현재 정의되지 않음(단일 화면).
- 딥링크 확장 여지는 12절(오픈 이슈) 참조.

## 4. 화면 레이아웃

상단 헤더 영역과 하단 일정표 영역으로 구성된다.

```text
┌─────────────────────────────────────────────────────────────┐
│ 진행현황(WBS)                                                  │  ← 제목
│ 과제: 표준 목록 생성 및 상세 일정 비교 모델 개발               │  ← 과제명
│ 각 업무의 세부 일정과 진행 상태를 월/주 단위로 표시           │  ← 설명
│                                                               │
│ [범례] ■ 완료   ■ 진행 중   ▢ 진행 예정      전체 진행률 62% │  ← 범례 + 진행률
├──────────┬───────────────┬──────────┬─────────────────────────┤
│ 구분     │ 세부 업무     │ 진행상태 │  2월        3월      … │  ← 시간축 헤더(월)
│          │               │          │ W1 W2 W3 W4 W1 W2 …    │  ← 시간축 헤더(주차)
├──────────┼───────────────┼──────────┼─────────────────────────┤
│ 데이터   │ 입력 문서 수집 │ 완료     │ ██████                  │
│ 준비     │ 표준 목록 초안 │ 진행 중  │      ████████░░         │
├──────────┼───────────────┼──────────┼─────────────────────────┤
│ 모델     │ 비교 로직 설계 │ 진행 예정│                  ░░░░   │
│ 개발     │ …             │ …        │                         │
└──────────┴───────────────┴──────────┴─────────────────────────┘
```

- 좌측 고정 열: `구분 / 세부 업무 / 진행상태`. 그리드 가로 스크롤 시에도 고정(sticky)되어 행의 정체성을 유지한다.
- 우측 스크롤 영역: 월/주차 그리드. 기간이 길면 가로로 확장되며 시간축 헤더는 상단 고정.
- 반응형: 좁은 화면에서는 좌측 열을 축약하거나 그리드를 우선 노출. 화면폭이 매우 좁을 때는 행 단위 요약(막대 시작~종료 + 상태)으로 대체 가능.

## 5. 컴포넌트 인벤토리

| 컴포넌트 | 역할 | 비고 |
| --- | --- | --- |
| 페이지 헤더 | 제목·과제명·설명 표시 | 정적 텍스트 |
| 범례(Legend) | 상태 3종의 색상 키 안내 | 완료/진행 중/진행 예정 |
| 전체 진행률 표시 | 과제 단위 종합 진행률(수치+막대) | 산출식은 7절 참조 |
| WBS 일정표(Gantt Grid) | 핵심 컴포넌트. 행×시간축 매트릭스 | 아래 4개 하위 요소로 구성 |
| ├ 좌측 정보 열 | 구분·세부 업무·진행상태 | 그룹 헤더(구분) 병합 표시 |
| ├ 시간축 헤더 | 월 행 + 주차 행(2단) | 상단 고정 |
| ├ 일정 막대(Bar) | 업무의 시작~종료 구간 | 상태별 색상 |
| └ 그리드 셀 | 주차 단위 배경 칸 | 오늘 위치 강조선 |
| 마일스톤 마커 | 생성 일정 마일스톤을 그리드 위에 표기 | 제출 마일스톤 유형 A/B |
| 그룹 접기/펼치기 | 구분 단위로 행 묶음 토글 | |
| 진행률 배지/칩 | 행별 진행 상태 라벨 | 텍스트+색상 |
| 빈 상태(Empty) | 업무가 없을 때 안내 | 10절 참조 |

색상 규칙(범례와 동일):

| 상태 | 표현 |
| --- | --- |
| 완료 | 파란 막대(채움) |
| 진행 중 | 초록 막대(채움) |
| 진행 예정 | 회색 막대 또는 빈 일정 |

## 6. 데이터 모델

WBS 행은 Task 세부 업무(type=task)에서 1:1로 파생된 WBSItem이다. Task가 업무 원장으로 일정·담당·상태·진행률을 보유하고, WBS는 그 값을 시간축으로 투영·집계만 한다(따로 저장하지 않음). 정본 정의는 `../data-model.md`, 상태 값 정의는 `../status-taxonomy.md`를 따른다.

WBS 렌더링에 쓰이는 필드(개념 스키마):

```text
WbsItem  (모든 필드는 대응 Task 세부 업무에서 투영)
├─ id            : 대응 세부 업무 식별자
├─ group         : 구분(상위 카테고리, 그룹 헤더로 묶임)
├─ title         : 세부 업무명
├─ status        : 완료 | 진행 중 | 진행 예정
├─ owner         : 담당자
├─ startDate     : 시작일(막대 좌측 경계 산출용)
├─ endDate       : 종료일(막대 우측 경계 산출용)
├─ progress      : 진행률(0~100)  ── 없으면 status로 대체 산출
└─ order         : 그룹 내 정렬 순서
```

- 시간축(월/주차)은 저장 데이터가 아니라 전체 업무의 최소 `startDate`~최대 `endDate`에서 파생 계산한다.
- `group`은 대응 Task 세부 업무의 상위 구분(category)명을 비정규화한 값으로, 좌측 열에서 세로 병합되어 그룹 헤더처럼 보인다.

## 7. 상태 및 전이

### 7.1 업무 상태(행 단위)

세 가지 상태를 가지며, 정상 흐름은 단방향으로 진행된다.

```text
진행 예정 ──▶ 진행 중 ──▶ 완료
   ▲            │
   └────────────┘   (예외: 재개/롤백 시 역전이 가능)
```

| 상태 | 그리드 표현 | 의미 |
| --- | --- | --- |
| 진행 예정 | 회색 막대 / 빈 일정 | 아직 착수하지 않음 |
| 진행 중 | 초록 막대 | 착수했고 완료 전 |
| 완료 | 파란 막대 | 마무리됨 |

상태 값의 정본 정의·전이 규칙은 `../status-taxonomy.md`를 우선한다. 본 화면은 이를 색상으로 반영하는 표현 계층이다.

### 7.2 셀 상태(그리드 칸 단위)

각 주차 셀은 해당 업무 막대와의 관계로 결정된다.

| 셀 상태 | 조건 | 표현 |
| --- | --- | --- |
| 막대 없음 | 주차가 업무 기간 밖 | 빈 배경 |
| 막대 시작 | 주차 = startDate 주차 | 막대 좌측 라운드 |
| 막대 중간 | 기간 내부 주차 | 막대 채움 |
| 막대 끝 | 주차 = endDate 주차 | 막대 우측 라운드 |
| 오늘 포함 | 주차가 현재 주 | 오늘 강조선 오버레이 |

### 7.3 진행률 산출

전체 진행률과 (선택적) 행 진행률의 계산 규칙:

- 행에 `progress` 값이 있으면 그 값을 그대로 사용.
- 없으면 상태로 환산: 완료=100%, 진행 중=50%, 진행 예정=0%.
- 전체 진행률 = 모든 업무 진행률의 평균(단순 평균). 가중치가 필요하면 12절 오픈 이슈 참조.

```text
전체 진행률 = Σ(업무별 progress) / (업무 수)
예) 완료 3, 진행 중 2, 진행 예정 1  →  (100×3 + 50×2 + 0×1) / 6 ≈ 66.7%
```

## 8. 액션 · 인터랙션

현재 범위에서 WBS는 조회 중심 화면이며, 주요 인터랙션은 탐색과 확인에 한정된다.

| 인터랙션 | 동작 |
| --- | --- |
| 가로 스크롤 | 기간이 길 때 시간축을 좌우로 이동. 시간축 헤더·좌측 열 고정 유지 |
| 세로 스크롤 | 업무 수가 많을 때 행 목록 이동. 시간축 헤더 상단 고정 |
| 막대/행 호버 | 툴팁으로 시작~종료일·상태·진행률 표시(구현 시) |
| 범례 확인 | 색상-상태 매핑 참조(정적) |
| 오늘로 이동 | 진입 시 현재 주차가 보이도록 자동 스크롤(기본 on) |
| 그룹 접기/펼치기 | 구분 단위로 하위 행을 열고 닫음 |
| 마일스톤 확인 | 그리드 위 마일스톤 마커로 제출 기준 시점 표시 |

- 편집(막대 드래그로 일정 변경 등)은 도입하지 않는다. 일정 변경은 Task 편집 흐름을 통한다(9절).
- 시간축은 월/주 2단으로 고정한다(일 단위는 향후). 상태 필터와 구분(그룹) 접기/펼치기를 제공한다.

## 9. 사용자 흐름

### 9.1 진행 현황 점검(주 흐름)

```text
1차 메뉴에서 WBS 선택
  → /wbs 진입, 일정표·전체 진행률 로드
  → 범례로 색상 의미 확인
  → 지연·임박 업무를 시간축에서 스캔
  → 필요 시 세부 확인(호버 툴팁)
```

### 9.2 업무 상세로의 이동(연계 흐름)

```text
WBS에서 특정 업무 확인
  → Task 페이지에서 해당 업무 열람/수정
  → 상태·일정 변경 반영
  → WBS로 돌아와 막대·진행률 갱신 확인
```

일정과 진행률의 원천 편집은 Task/Work에서 이뤄지고, WBS는 그 결과를 시각화한다.

## 10. 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 업무가 하나도 없음 | 빈 상태 안내, 전체 진행률 표시 보류 또는 0% 처리 |
| 시작일/종료일 누락 | 해당 막대 미표시 또는 물음표 마커, 좌측 열에는 노출 |
| 종료일 < 시작일 | 데이터 오류로 간주, 경고 표시(막대는 시작 주차만) |
| 기간이 과도하게 길다 | 가로 스크롤로 처리(시간축은 월/주 고정) |
| 오늘이 전체 기간 밖 | 오늘 강조선 생략 |
| 진행률과 상태 불일치 | 예: 상태 완료인데 progress<100 — 상태 색상 우선, 진행률은 값 그대로 |
| 단일 주차짜리 업무 | 한 칸 막대(좌우 라운드 동시 적용) |
| 모두 진행 예정 | 전체 진행률 0%, 그리드는 회색/빈 일정만 |

## 12. 오픈 이슈

- 시간축: 월/주 2단 고정, 일 단위는 향후. — 결정 완료.
- 전체 진행률: 단순 평균 유지(가중치 미적용). — 결정 완료.
- 막대 직접 편집: 미도입(조회 전용, 편집은 Task). — 결정 완료.
- 오늘 자동 스크롤 기본 on, 상태 필터 제공. — 결정 완료.
- 그룹(구분) 접기/펼치기: 지원. — 결정 완료.
- 마일스톤 마커: 표기함. — 결정 완료.
- 업무 상세 딥링크(`/wbs/:id`): 미도입(Task/Work 상세로 이동). — 결정 완료.
