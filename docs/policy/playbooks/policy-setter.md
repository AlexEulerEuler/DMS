# 플레이북: 개발 정책 세터

## 요약

특정 area의 개발 정책(코딩 스타일·테스트 규약·금기)을 세우는 절차. 핵심 원칙:
**문서에만 있는 규칙을 만들지 마라** — 규칙마다 기계 강제 수단(린터 설정·CI)을 함께 산출한다.

## 절차

1. **입력**: 대상 area의 스택과 기존 코드(있다면). 표준 가이드라인 후보를 조사한다.
2. **정책 문서**: `docs/policy/area/<area>.md`를 [템플릿](../../templates/policy.md)으로 작성 —
   코딩 스타일, 테스트 명령·커버리지 기준, 디렉터리 규칙, 금기(하지 말 것), 도메인 규칙.
   각 규칙에 강제 수단을 병기하고, 기계 강제가 불가능한 규칙은 '절차 규율'로 명시한다.
3. **강제 수단**: 린터·포매터 설정 파일(eslint/ruff/prettier 등)을 실제로 추가하고
   quality CI에 연결한다. 검수 체크리스트(20-review-policy)에 반영할 항목이 있으면 개정 PR을 분리한다.
4. **진입점 갱신**: 해당 앱 `AGENTS.md`에 정책 문서 링크를 추가한다.
5. **산출**: 정책 문서 PR은 `docs/policy/**` 대상이므로 **T2 경로(24h 쿨다운)**다.

## 완료 기준

에이전트가 해당 area에서 `/dms-work`를 수행할 때 이 정책만 읽으면 스타일·테스트 기준을
스스로 준수할 수 있고, 위반이 CI에서 기계적으로 걸린다.
