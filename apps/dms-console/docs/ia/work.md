# Work

## 1. 개요

Work는 프로젝트 진행에 필요한 개발 작업을 등록하고, 담당자에게 배정하며, 진행 상태를 추적하는 모듈이다. 동일한 작업 집합을 두 가지 방식으로 다룬다. Backlog는 테이블 기반으로 전체 작업을 목록·검토하는 관리 뷰이고, Kanban은 상태별 열에 카드를 배치해 진행 흐름을 시각적으로 관리하는 실행 뷰다.

Work의 작업 항목은 상위 계획(Task/WBS)의 실행 단위에 해당하며, Issues(GitHub)에서 넘어온 요청·리스크를 실제 개발 작업으로 전환하는 접점이다. 에이전트 구현 작업은 Agent 기획과 연결해 구현 대상을 명시할 수 있다.

## 2. 내비게이션 위치

- 1차 메뉴: Work (Overview, Task, WBS, Issues, Work, Agent 중 다섯 번째)
- 2차 메뉴(좌측 사이드바): Backlog, Kanban
- 선택된 2차 메뉴는 좌측 사이드바에서 강조 표시된다.

```text
Work
├── Backlog   (테이블 뷰)
└── Kanban    (보드 뷰)
```

작업 등록(new)과 작업 상세(:workId)는 사이드바에 노출되지 않는 파생 화면으로, Backlog·Kanban에서 진입한다.

## 3. 하위 구조 / 라우트

```text
/work/backlog     Backlog 테이블
/work/kanban      Kanban 보드
/work/new         작업 등록
/work/:workId     작업 상세
```

- 기본 진입점은 `/work/backlog`이며, `/work`로 접근 시 Backlog로 리다이렉트한다.
- `/work/new`는 Backlog·Kanban 어디에서 진입해도 동일한 등록 폼을 사용한다. 등록 완료 후에는 직전 뷰로 복귀한다.
- `/work/:workId`는 하나의 작업을 상세·편집하는 화면으로, 유효하지 않은 `workId`는 오류 상태로 처리한다(10장 참고).

## 4. 화면 레이아웃

### 4.1 Backlog

- 헤더: 제목(Backlog), 설명("개발자에게 할당할 작업 관리"), 요약(총 작업 수)
- 액션 바: 작업 등록 버튼, 상태 필터, (선택) 검색 입력
- 본문: 작업 테이블. 행 단위로 작업 하나를 표시하며 각 행에서 View로 상세 진입
- 컬럼: 작업명 · 담당 · 진행상태 · 진행 일정(시작~종료)

### 4.2 Kanban

- 헤더: 제목(Kanban), 상태별 카드 수 요약
- 본문: 4개 상태 열(Todo / In Progress / Review / Done)이 가로로 배치된 보드
- 각 열: 열 제목, 카드 수, 세로 카드 목록
- 카드: 제목 · 담당자 · 일정 · 상태 · 연결 이슈. 클릭 시 상세로 진입, 드래그로 열 이동

### 4.3 작업 등록 / 상세

- 등록(new): 단일 폼(작업명, 담당, 상태, 시작·종료일, 설명, 연결 이슈, 연결 에이전트) + 저장·취소
- 상세(:workId): 상단 요약(작업명·상태 배지·담당·일정), 본문 상세 필드, 연결 이슈·연결 에이전트 링크, 하단 액션(저장, 삭제, 목록으로)

## 5. 컴포넌트 인벤토리

| 화면 | 컴포넌트 | 설명 |
|------|----------|------|
| Backlog | 요약 카운터 | 총 작업 수 표시 |
| Backlog | 상태 필터 | 진행예정/진행중/리뷰중/완료로 목록 필터 |
| Backlog | 작업 테이블 | 작업명·담당·진행상태·진행 일정 컬럼 |
| Backlog | View 액션 | 행에서 상세로 진입 |
| 공통 | 작업 등록 버튼 | `/work/new`로 이동 |
| Kanban | 상태 열 | Todo/In Progress/Review/Done |
| Kanban | 작업 카드 | 제목·담당자·일정·상태·연결 이슈 |
| Kanban | 드래그 핸들 | 열 간 이동으로 상태 전이 |
| new | 작업 폼 | 데이터 모델 필드 입력 |
| :workId | 상태 배지 | 현재 진행 상태 표시 |
| :workId | 연결 링크 | 연결 이슈·연결 에이전트로 이동 |

## 6. 데이터 모델

작업 항목(WorkItem)의 핵심 필드는 다음과 같다. 전체 정의와 다른 모듈과의 관계는 [data-model.md](./data-model.md)를 따른다.

| 필드 | 설명 |
|------|------|
| id | 작업 식별자, 라우트의 `:workId` |
| title | 작업명 |
| owner | 담당자 |
| status | 진행 상태(7장) |
| startDate / endDate | 진행 일정(시작·종료) |
| description | 작업 설명 |
| linkedIssue | 근거 이슈(Issue 참조, N:1) |
| linkedAgent | 참조 에이전트 기획(Agent 참조) |

- `linkedIssue`, `linkedAgent`는 각각 0..1 관계이며 없을 수 있다.
- Backlog·Kanban은 동일한 WorkItem 집합을 공유한다. 표현만 다를 뿐 데이터는 하나다.

## 7. 상태 및 전이

Work 상태는 진행예정 · 진행중 · 리뷰중 · 완료 네 단계다. 상태 정의의 단일 출처는 [status-taxonomy.md](./status-taxonomy.md)이며, 여기서는 Kanban 열과의 매핑만 다룬다.

### 7.1 칸반 열 ↔ 작업 상태 매핑

| Kanban 열 | Work 상태 |
|-----------|-----------|
| Todo | 진행예정 |
| In Progress | 진행중 |
| Review | 리뷰중 |
| Done | 완료 |

- 카드를 다른 열로 옮기면 대응하는 작업 상태가 갱신되고, Backlog 테이블의 진행상태에도 즉시 반영된다.
- 반대로 Backlog·상세에서 상태를 바꾸면 Kanban의 해당 카드가 매핑된 열로 이동한다. 두 뷰는 같은 상태 값을 양방향으로 공유한다.

### 7.2 표준 전이

```text
진행예정 → 진행중 → 리뷰중 → 완료
```

- 되돌리기(예: 리뷰중 → 진행중)는 리뷰 반려 시 허용한다.
- 완료 이후 재개가 필요하면 진행중으로 되돌린다.

## 8. 액션 · 인터랙션

- 작업 등록: 작업 등록 버튼 → `/work/new` → 저장 시 목록에 추가되고 직전 뷰로 복귀
- 상세 열람: Backlog의 View 또는 Kanban 카드 클릭 → `/work/:workId`
- 상태 변경: Kanban 드래그로 열 이동, 또는 상세·Backlog에서 상태 선택
- 필터: Backlog 상태 필터로 목록 좁히기
- 연결 이동: 상세·카드의 연결 이슈/연결 에이전트에서 해당 모듈로 이동
- 삭제: 상세 화면에서 작업 삭제

## 9. 사용자 흐름

### 9.1 이슈에서 작업 생성

```text
Issues 이슈 확인
→ Work 작업 등록(연결 이슈 지정)
→ 담당자·일정 입력
→ Backlog에 등록
→ Kanban에서 진행 관리
```

### 9.2 진행 관리

```text
Kanban Todo 카드 확인
→ 담당자 착수 시 In Progress로 이동
→ 작업 완료 시 Review로 이동
→ 검토 통과 시 Done
→ WBS/Task 진행 상태에 반영
```

### 9.3 에이전트 구현 작업 연결

```text
작업 등록
→ 연결 에이전트(기획) 지정
→ Agent 기획을 참조해 구현·진행
→ 작업 상태 갱신
```

## 10. 엣지 케이스

- 존재하지 않는 `workId` 진입: 오류 안내와 목록 복귀 경로 제공
- 작업 0건: Backlog 빈 목록·Kanban 빈 열에 안내 문구와 작업 등록 유도
- 담당·일정 미지정: 등록은 허용하되 목록에서 미지정 상태로 구분 표시
- 연결 이슈/에이전트가 삭제된 경우: 연결 표시를 비활성화하고 끊어진 링크로 처리
- 종료일이 시작일보다 빠른 입력: 저장 전 유효성 경고
- 상태와 칸반 열 불일치(데이터 이상): 상태 값을 기준으로 열 위치를 재정렬

## 12. 오픈 이슈

- 카드 이동 시 알림: 현재 없음(단일 사용자). — 결정 완료.
- 작업 상태 변경은 WBS/Task 진행률에 자동 반영. — 결정 완료.
- 작업은 이슈 1개·에이전트 1개(N:1)로 단수 연결한다. 이슈–여러 에이전트는 여러 작업으로 표현. — 결정 완료(5.2).
- Backlog 정렬·검색은 리스트 공통 규약([foundation.md](./foundation.md) 8.1)을 따른다(등록일 최신순, 상태 필터, 제목 검색). — 결정 완료.
