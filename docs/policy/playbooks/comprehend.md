# 플레이북: 기획 컴프리헨더

## 요약

자유 형식 기획안을 에이전트가 착수 가능한 상태(기획 정본 + 초기 골격 + 작업 이슈 초안)로
변환하는 절차. 저빈도 부트스트랩 작업이라 슬래시 커맨드가 아닌 플레이북으로 둔다 —
에이전트에게 이 문서를 지목해 위임한다.

## 절차

1. **입력 확정**: 기획안(md/텍스트)을 받는다. 대상 앱(apps/frontend·backend 등)을 확인한다.
2. **기획 정본 작성**: 기획안을 `apps/<앱>/docs/product/`(앱 종속) 또는 `docs/product/`(전사)에
   메타 정책 준수(frontmatter, 요약 3~5줄, 템플릿)로 정리한다. 원문 기획안의 모호한 지점은
   임의 보완하지 말고 "미확정" 섹션으로 명시한다.
3. **용어 추출**: 새 도메인 용어를 [docs/glossary.md](../../glossary.md)에 추가한다(재정의 금지 원칙 확인).
4. **초기 골격**: 필요한 최소한의 디렉터리·모듈 스캐폴드와 각 모듈 README를 만든다.
   실제 구현은 하지 않는다 — 골격은 작업 분해의 캔버스다.
5. **작업 분해**: 기획을 반나절~수일 규모의 이슈 초안으로 분해해 `gh issue create`한다
   (이슈 폼 필드 전부, `by:agent` 라벨). 큰 덩어리는 에픽+하위 체크리스트로.
   `docs/plan/tasks.yaml`에 대응하는 T-노드를 추가하고 이슈에 `task:T-###`를 단다.
6. **산출**: 2~5는 하나의 PR(`Refs #발원이슈`)로 제출한다. **ready 라벨은 달지 않는다.**

## 완료 기준

사람이 이 PR과 이슈들만 검토·승인(`ready`)하면 데브 에이전트가 즉시 착수 가능한 상태.
