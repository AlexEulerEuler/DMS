# DMS 원칙

이 문서는 모든 정책·기획·작업 정의가 따라야 하는 최상위 원칙이다. 1페이지를 넘기지 않는다.

## 1. 정본은 두 곳뿐이다

- **실행 상태**(누가 무엇을 하고 있고 무엇이 끝났나)의 정본은 **GitHub**다 — 이슈, assignee, PR, 머지.
- **정의와 규칙**(무엇을 왜 만들고 어떻게 일하나)의 정본은 **리포 안의 문서**다.
- 같은 사실을 두 곳에 쓰지 않는다. 대시보드·캐시·요약은 전부 파생물이며, 어긋나면 정본이 이긴다.

## 2. 클론이 곧 작업 환경이다

개발자(사람이든 에이전트든)에게 필요한 것은 `git clone` + `gh` CLI + 코딩 에이전트뿐이다.
어떤 서버·MCP·대시보드도 작업의 필수 경로에 두지 않는다. 대시보드가 꺼져 있어도 개발은 진행된다.

## 3. 문서 작성이 곧 위임이다

에이전트에게 일을 시키는 방법은 문서를 쓰는 것이다 — 작업 정의(완료 조건·비범위), 정책, 검수 기준.
작업 품질의 상한은 정의 품질이 결정한다. 에이전트가 반복해서 잘못하면 코칭이 아니라 문서를 고친다.

## 4. 선의에 기대는 규칙은 규칙이 아니다

규칙마다 기계 강제 수단(CI·브랜치 보호·린터)을 함께 만든다. 기계로 강제할 수 없는 규칙은
그 사실을 명시하고(잔여 위험), 감사로 보완한다. 유일하게 신뢰하는 기계적 경계는
CI가 fresh 컨텍스트에서 생성하는 required check다.

## 5. 사람이 반드시 하는 일

작업 정의와 `ready` 승인, 정책 문서 유지, PR 머지(T1 이상), 표본 감사, 예외 처리.
이것이 줄어드는 것이 아니라, 이것*만* 남는 것이 목표다.
