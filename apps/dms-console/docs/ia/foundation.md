# Foundation

DMS Console의 공통 앱 셸과 글로벌 내비게이션 기반 문서. 모든 1차 메뉴 화면이 공유하는 전역 레이아웃, 내비게이션 규칙, 라우팅, 공통 상태, 접근성 기준을 여기서 단일 기준으로 정의한다. 개별 메뉴의 상세 IA는 각 문서로 링크만 둔다.

## 1. 개요

- 목적: 콘솔의 뼈대를 정의한다. 상단 바·좌측 사이드바·본문 영역으로 이루어진 셸, 1차/2차 내비게이션 규칙, URL·라우팅 규약, 그리고 빈/로딩/에러 같은 공통 상태를 한곳에 모아 모든 화면이 같은 규칙을 따르게 한다.
- 범위: 화면 공통 프레임과 내비게이션·라우팅·상태·접근성. 각 1차 메뉴(Overview, Task, WBS, Issues, Work, Agent)의 화면 구성·데이터·플로우는 다루지 않고 해당 문서로 위임한다.
- 주 사용자: 콘솔을 구현하는 개발자, 화면을 설계하는 기획·디자인 담당자. 새 화면을 추가할 때 이 문서의 규칙을 준수 기준으로 삼는다.

콘솔은 하나의 셸 안에서 6개 모듈을 오가는 단일 페이지 구조를 기본으로 한다. 셸은 고정이고, 본문 영역만 라우트에 따라 교체된다.

## 2. 전역 레이아웃

콘솔의 모든 화면은 상단 바, 좌측 사이드바, 본문 영역 3분할 셸을 공유한다.

```text
┌──────────────────────────────────────────────────────────────┐
│ 상단 바                                                        │
│  (좌) 서비스명 DMS Console                                     │
│  (중) 1차 메뉴 6: Overview · Task · WBS · Issues · Work · Agent │
│  (우) 설정                                                    │
├───────────────┬──────────────────────────────────────────────┤
│ 좌측 사이드바  │ 본문 영역                                     │
│               │                                               │
│ 현재 1차 메뉴  │  페이지 헤더(제목 · 설명 · 액션)              │
│ 의 2차 메뉴    │  ─────────────────────────────────────────── │
│               │  콘텐츠(목록 · 상세 · 폼 · 보드 등)           │
│ (선택=파란     │                                               │
│  배경)        │  ← 라우트에 따라 이 영역만 교체됨             │
└───────────────┴──────────────────────────────────────────────┘
```

- 상단 바: 콘솔 전역에서 항상 표시. 서비스명(좌), 1차 메뉴(중), 도구(우)로 구성되며 스크롤 시 고정된다.
- 좌측 사이드바: 현재 선택된 1차 메뉴에 종속된 2차 메뉴만 표시. 1차 메뉴가 바뀌면 사이드바 내용도 함께 바뀐다. 2차 항목이 하나뿐인 메뉴(WBS 등)에서도 사이드바를 그대로 표시한다.
- 본문 영역: 라우트에 대응하는 화면을 렌더. 상단 바와 사이드바는 유지한 채 이 영역만 교체된다. 긴 콘텐츠는 본문 영역이 세로 스크롤을 담당하고 셸 프레임은 고정한다.

## 3. 글로벌 내비게이션(1차)

상단 바 중앙의 1차 메뉴는 콘솔의 최상위 진입점이다.

```text
DMS Console
├── Overview   프로젝트 문서 허브
├── Task       업무 정보구조
├── WBS        일정·진행 현황
├── Issues     이슈·리스크·요청 관리
├── Work       개발 작업 관리
└── Agent      에이전트 기획
```

- 항목 수: 6개 고정. 순서는 위 나열 순서를 유지한다.
- 선택 상태: 현재 위치한 모듈의 1차 메뉴 항목을 활성 표시한다. 활성 판정은 URL의 첫 세그먼트 기준(예: `/work/kanban`이면 Work 활성).
- 진입 동작: 항목 클릭 시 해당 모듈의 기본 라우트로 이동한다(6·7절의 기본 라우트 참조).
- 서비스명(로고) 클릭: 앱 루트(`/`)로 이동하며, 루트는 Overview로 연결된다.

각 1차 메뉴의 세부 화면은 다음 문서에서 다룬다: [overview.md](./overview.md), [task.md](./task.md), [wbs.md](./wbs.md), [issues.md](./issues.md), [work.md](./work.md), [agent.md](./agent.md).

## 4. 좌측 내비게이션(2차)

좌측 사이드바는 현재 1차 메뉴에 종속된 2차 메뉴를 표시한다.

- 종속성: 사이드바 내용은 선택된 1차 메뉴에 따라 결정된다. Overview는 "프로젝트 문서" 하위 12개 문서, Task는 IA, Work는 Backlog·Kanban 등으로 각 모듈이 자기 2차 목록을 갖는다.
- 선택 상태: 선택된 2차 항목은 파란 배경으로 강조한다. 색상·대비 등 시각 규칙은 10절과 [status-taxonomy.md](./status-taxonomy.md)의 색 사용 기준을 따른다.
- 그룹 라벨: 사이드바 안의 섹션 헤더(예: "프로젝트 문서")는 링크가 아니라 그룹 제목이다. 실제 이동은 그 하위 항목에서 일어난다.
- 단일 항목 모듈: 2차 메뉴가 하나뿐이어도(WBS 등) 사이드바를 그대로 표시한다. 항목이 하나여도 숨기지 않아 내비게이션 구조가 일관된다.

3depth 이상(목록에서 진입하는 상세·편집)은 사이드바에 항목으로 나열하지 않는다. 상세·편집은 본문 영역에서 열리고, 위치는 브레드크럼/뒤로 가기로 표현한다.

## 5. 내비게이션 Depth 정책

콘솔은 최대 4depth 구조를 사용한다.

```text
1depth  1차 메뉴 6개                       상단 바
2depth  좌측 사이드바의 2차 메뉴           사이드바
3depth  목록에서 진입하는 상세·편집 화면   본문 영역(라우트 파라미터)
4depth  상세 화면 내부 탭                  본문 영역(상세 내 탭)
```

- 1depth: 상단 바의 6개 모듈. 항상 노출.
- 2depth: 사이드바 항목. 선택 시 파란 배경.
- 3depth: 목록 화면에서 개별 항목을 열거나 새로 만드는 화면(예: `/issues/:issueId`, `/work/new`). 사이드바에는 노출하지 않고 본문에서 전개한다.
- 4depth: 상세 화면 내부의 탭. 현재 범위에는 쓰는 화면이 없고, 향후 Agent 상세 고도화(flow·플레이그라운드) 시 도입한다. depth의 최하단이며 그 아래로 더 내려가지 않는다.

원칙: 화면 깊이는 4depth를 넘기지 않는다. 더 깊은 정보는 탭·패널·모달로 수용하고 새 라우트 depth를 늘리지 않는다.

## 6. 전체 라우트 맵

콘솔 전 구간의 라우트를 모듈별로 정리한다. 각 라우트의 화면 세부는 해당 모듈 문서를 참조한다.

```text
/                                      → /overview 로 이동(앱 루트)

Overview  → overview.md
/overview                              프로젝트 개요(기본)
/overview/glossary                     도메인 용어
/overview/pipeline                     전체 파이프라인
/overview/modules                      서비스 모듈
/overview/api                          API 엔드포인트
/overview/cli                          CLI 명령어
/overview/folder-structure             폴더 구조
/overview/data                         주요 데이터
/overview/tech-stack                   기술 스택
/overview/env                          환경 변수
/overview/runbook                      실행 방법
/overview/limitations                  알려진 한계

Task  → task.md
/task                                  → /task/ia 로 이동(기본)
/task/ia                               업무 IA 트리

WBS  → wbs.md
/wbs                                   진행 현황(일정표)

Issues  → issues.md
/issues                                이슈 목록(기본)
/issues/new                            이슈 등록
/issues/:issueId                       이슈 상세

Work  → work.md
/work/backlog                          작업 백로그(기본)
/work/kanban                           작업 칸반 보드
/work/new                              작업 등록
/work/:workId                          작업 상세

Agent  → agent.md
/agents                                에이전트 목록(기본)
/agents/new                            에이전트 등록
/agents/:agentId                       에이전트 상세(기획 텍스트)

(상세 하위 탭 /agents/:agentId/... 은 향후 Agent 고도화 시 추가)
```

- 각 모듈의 기본 라우트(위 "기본" 표기)는 1차 메뉴 클릭 시 도달하는 진입점이다.
- Agent 모듈의 1차 메뉴 라벨은 "Agent"이나 라우트 경로는 복수형 `/agents`를 사용한다. 라벨과 경로가 다른 유일한 사례다.
- 상세 하위 탭(4depth)은 현재 없으며, Agent 상세 고도화 시 추가된다.

## 7. URL·라우팅 규칙

- 경로 표기: 소문자, 단어 구분은 하이픈(kebab-case). 예) `/overview/folder-structure`, `/overview/tech-stack`.
- 세그먼트 의미: 첫 세그먼트 = 1차 메뉴(모듈), 이후 세그먼트 = 2차 메뉴 또는 상세/탭.
- 동적 세그먼트: 상세 화면은 `:id` 형태의 파라미터를 쓴다(`:issueId`, `:workId`, `:agentId`). 식별자는 라우트 파라미터로 전달한다.
- 기본 라우트 리다이렉트: 모듈 루트 진입 시 해당 모듈의 기본 화면으로 보낸다. `/` → `/overview`, `/task` → `/task/ia`, `/work` → `/work/backlog`. `/issues`와 `/agents`는 모듈 루트가 곧 목록 화면이라 별도 리다이렉트가 없다.
- 활성 상태 판정: 1차 메뉴 활성은 첫 세그먼트, 2차 메뉴 활성은 두 번째 세그먼트(또는 기본 라우트 일치)로 판정한다.
- 잘못된 경로: 정의되지 않은 경로나 존재하지 않는 `:id`는 항상 "없음"(not-found) 화면을 표시한다(8절). 자동 리다이렉트하지 않는다. (모듈 루트→기본 화면 리다이렉트는 위 항목의 정상 동작이며 예외)
- 라벨 대 경로 예외: Agent 라벨 ↔ `/agents` 경로가 유일한 예외다. 그 외에는 라벨과 경로 슬러그가 대응한다.

## 8. 공통 상태 셸(빈/로딩/에러/없음)

모든 콘텐츠 영역은 아래 공통 상태를 동일한 방식으로 표현한다. 개별 화면은 이 상태 셸을 재사용하고 문구만 맥락에 맞춘다.

```text
loading  → (성공) populated
         → (실패) error
empty    (성공이나 표시할 데이터가 없음)
error    → (재시도) loading
not-found (없는 라우트/식별자)
```

- loading: 데이터 로드 중. 콘텐츠 자리에 스켈레톤 또는 로딩 표시를 두고 상단 바·사이드바는 유지한다.
- empty: 로드는 성공했으나 항목이 0건. 빈 상태 안내와 함께 주 액션(등록 등) 진입점을 제공한다. 셸·내비게이션은 정상 동작.
- error: 로드·저장 실패. 오류 메시지와 "다시 시도"를 제공하고, 내비게이션은 계속 사용 가능하게 둔다. 부분 실패 시 실패 영역만 오류로 표시한다.
- not-found: 존재하지 않는 라우트/식별자. "없음" 화면을 표시한다(7절). 자동 리다이렉트하지 않는다.

상태 전이 중에도 셸(상단 바·사이드바)은 항상 렌더된 채 유지되어, 사용자가 다른 모듈로 빠져나갈 수 있게 한다.

### 8.1 리스트 공통 규약 (정렬·필터·페이지네이션)

목록형 화면(Issues, Work Backlog, Agent 목록 등)은 아래 규약을 공통으로 따르고, 개별 문서는 화면별 예외만 명시한다.

- 정렬: 기본은 등록일 최신순. 우선순위 축이 있는 목록(Issues)은 우선순위 → 등록일 순(우선순위 정의는 [status-taxonomy.md](./status-taxonomy.md) 9절).
- 필터: 상태 필터를 기본 제공(값은 status-taxonomy.md). Issues는 우선순위 필터를 추가 제공한다.
- 검색: 제목 기준 텍스트 검색을 기본으로 한다.
- 페이지네이션: 페이지 기반, 페이지 크기 25. 총 개수를 헤더 요약으로 노출한다(상세는 spec/README.md 5절).
- 빈/로딩/에러: 8절 공통 상태 셸을 재사용한다.

## 9. 상단 도구(설정)

상단 바 우측 도구 영역은 콘솔 전역 기능을 담는다.

- 설정: 콘솔 전반 설정 진입점. 위치는 상단 우측 고정.
- 전역성: 특정 모듈에 속하지 않고 어느 화면에서든 동일하게 노출된다.
- 배치: 도구 영역은 1차 메뉴와 시각적으로 구분되도록 상단 바 우측 끝에 정렬한다.

설정 항목의 구체 목록은 오픈 이슈로 둔다. 다국어는 현재 범위에서 다루지 않는다.

## 10. 접근성·반응형 기본

접근성 기본

- 키보드 이동: 상단 1차 메뉴 → 좌측 2차 메뉴 → 본문 순으로 논리적 포커스 이동을 보장한다. 모든 내비게이션·액션은 키보드로 도달·조작 가능해야 한다.
- 포커스 표시: 선택 상태(파란 배경)와 별개로, 키보드 포커스 링을 명확히 표시한다.
- 상태 전달: 선택/활성 상태를 색상에만 의존하지 않고 보조 표기(굵기·아이콘·aria 속성 등)로도 전달한다. 색 대비는 판독 가능한 수준을 유지한다.
- 의미 구조: 상단 바·사이드바·본문을 각각 내비게이션/메인 영역으로 구분해 스크린 리더가 구조를 인지하도록 한다.
- 상태 안내: 로딩/에러/빈 상태를 시각뿐 아니라 텍스트로도 전달한다.

반응형 기본

- 넓은 화면: 상단 바 + 좌측 사이드바 + 본문 3분할을 그대로 표시.
- 좁은 화면: 좌측 사이드바를 드로어(옆에서 밀려나오는 패널)로, 상단 1차 메뉴는 드로어/햄버거로 접는다. 본문 영역을 우선 확보하고 카드·그리드는 1열로 재배치한다.
- 고정 요소: 상단 바는 좁은 화면에서도 접근 가능하도록 유지하거나 축약 형태로 노출한다.
- 콘텐츠 스크롤: 스크롤은 본문 영역이 담당하고 셸 프레임은 고정한다.
