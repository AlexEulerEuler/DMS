# Issues

## 1. 개요

Issues는 프로젝트의 GitHub Issues를 콘솔에서 함께 보는 화면이다. 이슈·리스크·요청을 조회하고, 콘솔 안에서 개발 작업(Work)과 연결해 진행을 추적한다.

이슈의 정본(source of truth)은 GitHub다. 제목·본문·상태(open/closed)·라벨·담당자는 GitHub에서 오고, 콘솔은 이를 미러링해 보여 준다. 콘솔이 얹는 것은 두 가지 로컬 정보뿐이다: 우선순위와 연결된 Work 작업. 이슈의 완료·종료는 GitHub에서 닫히면(예: 연결된 PR 병합) 콘솔에 반영된다.

## 2. 내비게이션 위치

1차 메뉴 6개 중 네 번째다. Issues 선택 시 목록(`/issues`)이 기본으로 열린다. 상세·등록은 목록에서 진입하는 하위 화면이다.

## 3. 하위 구조 / 라우트

```text
Issues (/issues)
├── 이슈 목록        /issues
├── 이슈 등록        /issues/new       (GitHub에 이슈 생성)
└── 이슈 상세        /issues/:issueId
```

- `/issues` — 목록. GitHub 이슈를 테이블로 조회. 기본은 열린(open) 이슈, 닫힌(closed) 이슈는 아카이브 필터로 조회.
- `/issues/new` — 등록. 콘솔에서 작성하면 GitHub에 이슈로 생성된다(write-through).
- `/issues/:issueId` — 상세. 이슈 내용 조회, 우선순위·연결 작업 관리.

`:issueId`는 GitHub 이슈 번호에 대응한다. 없는 값이면 "없음" 화면(foundation 공통 상태). GitHub 연동이 끊기면 error 상태로 처리한다.

## 5. 컴포넌트 인벤토리

목록 뷰

- 페이지 헤더 — 제목, 설명, 요약(열린 이슈 수).
- 이슈 등록 버튼 — `/issues/new`(GitHub 생성).
- 이슈 테이블 — 열: 번호(#), 제목, 상태(open/closed), 우선순위, 라벨, 담당자. 행 클릭 시 상세.
- 상태 배지(open/closed) / 우선순위 배지 / 라벨 칩.
- 아카이브 필터 — 닫힌 이슈 표시 토글.
- 빈 상태 안내.

등록 뷰

- 입력 필드 — 제목, 본문, 우선순위(로컬), 담당자·라벨(GitHub).
- 생성 / 취소.

상세 뷰

- 헤더 — 이슈 번호·제목, 상태, GitHub 원문 링크.
- 메타 — 상태(open/closed), 우선순위, 담당자, 라벨, 등록일.
- 본문 — GitHub 원문.
- 연결 작업 영역 — 연결된 Work 항목들(1:N), 작업 생성·연결.

## 6. 데이터 모델

이슈는 GitHub에서 온 필드와 콘솔 로컬 오버레이로 나뉜다. 공통 정의는 [data-model.md](./data-model.md)를 따른다.

| 필드 | 출처 | 설명 |
| --- | --- | --- |
| `id`(이슈 번호) | GitHub | 이슈 식별자(라우트 `:issueId`) |
| `title` | GitHub | 제목 |
| `body` | GitHub | 본문 |
| `state` | GitHub | open / closed |
| `labels` | GitHub | 라벨 |
| `assignee` | GitHub | 담당자 |
| `createdAt` | GitHub | 등록일 |
| `htmlUrl` | GitHub | GitHub 원문 링크 |
| `priority` | 콘솔 | 우선순위(7절) — 로컬 오버레이 |
| `linkedWorkItems` | 콘솔 | 연결된 Work 작업들(1:N) |

`priority`와 `linkedWorkItems`만 콘솔이 저장하고 나머지는 GitHub가 정본이다. 관계·카디널리티는 [data-model.md](./data-model.md).

## 7. 상태 및 전이

이슈 상태는 GitHub의 open/closed를 그대로 따른다. 콘솔은 별도 상태 기계를 두지 않는다.

```text
open ──(GitHub에서 닫힘: 예) 연결 PR 병합)──▶ closed
closed ──(GitHub에서 재오픈)──▶ open
```

- open — 열린 이슈. 목록 기본 노출.
- closed — 닫힌(완료) 이슈. 기본 목록에서 숨기고 아카이브 필터로 조회.
- 완료 자동화: 연결된 작업/PR이 GitHub에서 이슈를 닫으면 콘솔의 상태도 closed로 반영된다. 콘솔이 자체 규칙으로 상태를 전이시키지 않는다.
- 세분화된 진행 단계가 필요하면 GitHub 라벨(예: `in progress`)을 쓰고, 콘솔은 이를 라벨로 표시한다(별도 상태 아님).

우선순위(`priority`)는 상태와 독립된 콘솔 로컬 축이며 값(긴급/높음/보통/낮음)의 정본은 [status-taxonomy.md](./status-taxonomy.md) 9절이다.

## 8. 액션·인터랙션

- 이슈 등록 — `/issues/new` → 제목·본문 입력 → GitHub에 이슈 생성. 우선순위는 콘솔에 저장.
- 이슈 조회 — 목록 행 클릭 → 상세.
- 우선순위·연결 관리 — 상세에서 우선순위 설정, Work 작업 생성·연결(`linkedWorkItems`).
- GitHub로 이동 — 상세의 원문 링크로 GitHub 이슈 열기.
- 아카이브 조회 — 필터로 닫힌 이슈 표시.
- 기존 이슈의 GitHub 필드(제목·본문·라벨·담당자·상태)는 콘솔에서 읽기 전용이며, 편집은 GitHub에서 한다(상세의 원문 링크). 콘솔이 쓰는 것은 새 이슈 생성(write-through), 우선순위, 연결 작업뿐이다.

## 9. 사용자 흐름

```text
Issues 목록(열린 이슈)
   → 상세에서 우선순위 지정
   → Work 작업 생성·연결(1:N)
   → Work Kanban에서 개발 진행
   → 연결 PR 병합 → GitHub 이슈 closed → 콘솔에 반영(아카이브)
```

이슈는 요청의 입구(GitHub), Work는 실행, WBS·Task는 계획·진척이다. 이슈 하나가 여러 작업으로 갈라질 수 있고(1:N), 작업 진척은 Work에서, 종료는 GitHub에서 확정된다.

## 10. 엣지 케이스

- GitHub 연동 실패 — error 상태(재시도). 목록·상세 로드 불가 시 안내.
- 이슈 0건 — 빈 상태 안내와 등록 유도.
- 존재하지 않는 `:issueId` — "없음" 화면.
- 닫힌 이슈 — 기본 목록에서 숨김. 아카이브 필터로 조회.
- 미연결 이슈 — `linkedWorkItems`가 비면 상세에서 작업 생성 진입점 노출.
- 우선순위 미지정 — 기본값 보통으로 표시.

## 12. 오픈 이슈

- GitHub 필드 편집 범위: 기존 이슈는 콘솔 읽기 전용, 편집은 GitHub에서. 콘솔은 생성(write-through)·우선순위·연결 작업만 쓴다. — 결정 완료.
- GitHub 동기화: 진입 시 API 조회 + 수동 새로고침(webhook 실시간은 향후). 인증은 GitHub 토큰/App을 서버에 설정(단일 사용자). — 결정 완료.
- 우선순위: 콘솔 로컬로만 관리(GitHub 라벨로 내보내지 않음). — 결정 완료.
- 작업↔이슈, 작업↔에이전트는 각각 단수 연결(N:1). 이슈는 여러 작업(1:N)을 통해 여러 에이전트와 간접 연관. — 결정 완료(5.2).
