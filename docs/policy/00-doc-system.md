# 문서 시스템 정책 (메타 정책)

## 요약

문서를 어디에, 어떤 형식으로, 어떤 절차로 쓰는지의 정본. 문서 작성이 곧 위임이므로
문서의 품질 게이트는 코드와 같다 — 모든 문서 변경은 이슈→브랜치→PR→검수를 탄다.

## 1. 배치 규칙

- 계층과 위치는 [docs/README.md](../README.md)의 표를 따른다. 새 문서는 반드시 그 표의 한 계층에 속해야 한다.
- 전사 규칙은 `docs/`, 특정 앱의 기획·스펙은 `apps/<앱>/docs/`에 둔다(코드와 같이 움직이는 문서는 앱 로컬).
- 정본 레지스트리(README.md)에 있는 주제를 다른 문서에 재서술하지 않는다 — 3줄 이내 요약 + 링크만.

## 2. 형식

- 모든 `docs/**` 문서는 YAML frontmatter를 갖는다: `layer`, `status`, `owner` (3개만 — `updated`류
  시간 필드는 두지 않는다. git log가 정본이며 대시보드가 파생 표시한다).
- `status`: `draft`(머지 가능, 경량 검토) → `approved`(승인권자 검토 필수) → `superseded`|`deprecated`.
  "리뷰 중" 상태는 없다 — 리뷰 중 = 그 문서를 고치는 PR이 열려 있는 상태(정본은 GitHub).
- 문서 최상단에 **요약 3~5줄** 섹션 필수 — 에이전트가 전문을 읽을지 판단하는 비용을 상수로 만든다.
- 링크는 상대 경로 + `.md` 확장자. `WD-`, `ADR-`, `T-` 같은 ID는 링크 밖 본문에도 텍스트로 남긴다(grep 추적).
- 템플릿: [templates/](../templates/) — 정책은 policy.md, 작업 스펙은 work.md, 결정은 decision.md.

## 3. 수명주기

- `approved` 문서의 내용 변경과 `draft→approved` 전이는 T2/T1 경로(티어 테이블 참조)의 검토를 받는다.
- 폐기는 삭제가 아니라 supersede다: frontmatter `status: superseded` + 대체 문서 링크,
  장문이면 `docs/archive/`로 이동하고 유입 링크를 함께 고친다(이동까지가 supersede 작업의 정의).
- ADR은 번호 연속·삭제 금지. 기각된 설계도 "왜 기각됐는지"를 남긴다.
- 에이전트는 `status: approved`가 아닌 문서를 정본으로 취급하지 않는다.

## 4. 충돌 해소

- 계층이 다르면 낮은 계층(원칙 > 정책 > 기획 > 작업 스펙)이 이긴다.
  단, **문서 형식·작성 절차에 관한 충돌은 계층과 무관하게 이 문서가 우선한다.**
- 충돌을 발견한 에이전트는 임의 해석하지 말고 `type:docs` 이슈를 연다(발견 시 의무).
- 코드 PR이 approved 스펙과 어긋나는 구현을 담으면: 같은 PR에서 스펙도 고치거나(검토 포함),
  스펙 쪽에 이슈를 연다. "코드 따로 문서 따로"는 금지다.

## 5. 컨텍스트 로딩 상한 (문서가 늘어도 독서량은 상수)

| 대상 | 상한 |
|---|---|
| 루트 AGENTS.md | 60줄 |
| 스코프 AGENTS.md | 150줄 |
| docs/principles.md | 1페이지(~50줄) |
| 이슈의 컨텍스트 문서 목록 | 5개 (절 앵커까지 지목 권장) |
| 링크 추적 깊이 | 2-hop 기본, 초과는 이슈 본문에 사유 기재 |

## 6. 관할 구분 (이중 발행 방지)

- 구현이 문서와 다르면 → **구현이 틀림** = `type:bug` (테스터 관할)
- 문서가 현실보다 낡았으면 → **문서가 틀림** = `type:docs` (문서 감사 관할)
