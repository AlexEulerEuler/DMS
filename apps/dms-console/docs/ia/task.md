# Task 페이지 IA

## 1. 개요

Task는 프로젝트 업무를 계층 구조로 정리하는 페이지다. 업무를 구분·세부 업무 계층으로 관리하며(중분류 계층은 스키마에 두되 현재 화면에는 노출하지 않는다), 여기서 등록·정리한 업무가 이후 WBS의 일정·진행률과 Work의 개발 작업으로 이어지는 출발점이 된다. 세부 업무는 담당자·기한·상태를 함께 갖는 태스크의 정의처다.

Task의 유일한 하위 화면은 IA(업무 정보구조) 트리다. 이 페이지는 "무엇을 해야 하는가"를 정의하는 계획 뷰이고, "언제 어떻게 진행되는가"는 WBS가, "누가 무엇을 개발하는가"는 Work가 담당한다. 즉 Task는 업무의 원장(source of truth) 역할을 하고, 상태·일정·실행 정보는 연관 모듈로 전개된다.

## 2. 내비게이션 위치

- 1차 메뉴: Task (상단 글로벌 내비게이션의 6개 항목 중 하나 — Overview, Task, WBS, Issues, Work, Agent)
- 2차 메뉴: IA (좌측 사이드바). Task 하위에는 현재 IA 화면만 존재한다.
- 선택 시 좌측 사이드바에서 해당 항목이 강조 표시된다.

```text
Task
└── IA        (/task/ia)
```

## 3. 하위 구조 / 라우트 맵

```text
/task            → /task/ia 로 리다이렉트(기본 진입)
/task/ia         업무 IA 트리(구분 · 중분류 · 세부 업무)
```

트리 자체는 별도 라우트로 분기하지 않고 단일 화면 안에서 펼침/접힘으로 탐색한다. 항목 추가와 상세 보기(View)는 화면 내 패널 또는 모달로 처리하는 것을 기본으로 한다.

## 4. 화면 레이아웃

```text
┌────────────────────────────────────────────────────────────┐
│ 제목: IA                                                     │
│ 설명: 업무 정보구조를 구분 · 중분류 · 세부 업무 계층으로 관리 │
├────────────────────────────────────────────────────────────┤
│ [상태 필터: 전체 | 완료 | 진행중 | 진행예정]   [항목 추가]   │
├────────────────────────────────────────────────────────────┤
│ ▸ 프로젝트 관리                                     [View]   │
│    · 기능 요건 정리 및 최종 산출물 정의             [상태]   │
│    · 데이터 수집 및 원천 데이터 정리                [상태]   │
│    · 데이터 관리 체계 및 버전 관리 기준 수립        [상태]   │
│ ▸ 표준 목록 생성                                    [View]   │
│    · 통합 표준 목록 생성                            [상태]   │
│    · 표준 목록 분류 체계 및 속성 정의               [상태]   │
│    · 통합 표준 목록 초안 자동 생성                  [상태]   │
│    · 협업 검토 기반 표준 목록 확정                  [상태]   │
│ ▸ 입력 문서 분석                                    [View]   │
│    · 입력 문서 구조 파악                            [상태]   │
│    · 요구 문서 및 제출 조건 추출                    [상태]   │
│    · 입력 문서–표준 목록 매칭 기준 수립             [상태]   │
│ ▸ 일정 생성                                         [View]   │
│    · 기준 일정 분석                                 [상태]   │
│    · 생성 일정/제출 마일스톤 기준 정의              [상태]   │
│    · 문서별 제출 일정 생성                          [상태]   │
└────────────────────────────────────────────────────────────┘
```

- 상단: 페이지 제목과 설명.
- 툴바: 좌측에 상태 필터, 우측에 항목 추가 액션.
- 본문: 구분(대분류) 노드를 접이식으로 나열하고, 각 구분 아래 세부 업무를 들여쓰기로 표시한다. 세부 업무 각 행에는 상태·담당자·기한이 표시된다.
- 원본 트리는 구분과 세부 업무 2단으로 구성되며, 중분류는 필요 시 구분과 세부 업무 사이에 삽입할 수 있는 선택 계층으로 둔다.

## 5. 컴포넌트 인벤토리

- 페이지 헤더: 제목(IA), 설명 텍스트
- 상태 필터: 전체 / 완료 / 진행중 / 진행예정 (단일 선택 탭 또는 세그먼트)
- 항목 추가 버튼: 새 구분·중분류·세부 업무 등록 진입점
- IA 트리
  - 구분(대분류) 노드: 펼침/접힘 토글, 라벨, View 액션
  - 중분류 노드(선택): 구분 하위 그룹핑 계층
  - 세부 업무 행: 라벨, 상태, 담당자, 기한(시작·종료)
- View 액션: 선택 노드의 상세/요약 표시
- 빈 상태 안내: 등록된 업무가 없을 때 노출
- 항목 추가/편집 폼: 라벨, 상위 노드, 상태, 담당자, 기한(시작·종료) 입력

## 6. 데이터 모델

Task IA 노드는 자기참조 트리로 표현한다. `type`으로 계층을 구분하고 `parentId`로 상하 관계를 연결한다.

```text
TaskIA
├── id            노드 식별자
├── type          category | group | task   (구분 | 중분류 | 세부 업무)
├── title         노드 라벨
├── parentId      상위 노드 id (구분은 null)
├── status        완료 | 진행중 | 진행예정   (세부 업무)
├── owner         담당자                     (세부 업무)
├── startDate     시작 기한                  (세부 업무)
├── endDate       종료 기한                  (세부 업무)
├── progress      진행률 0~100              (세부 업무)
├── order         동일 부모 내 정렬 순서
├── linkedWbsId   대응 WBS 항목 참조
├── createdAt
└── updatedAt
```

- 상태·담당자·기한·진행률은 세부 업무(type=task) 단위에 부여한다. 구분/중분류의 상태·진행률은 하위 세부 업무 집계로 자동 산출한다(수동 지정 아님).
- `linkedWbsId`는 세부 업무를 WBS 항목과 연결한다. 세부 업무 등록 시 대응 WBSItem이 1:1로 자동 파생되고, WBS는 이 세부 업무의 일정·진행을 시간축으로 투영만 한다(값은 Task가 보유). 이 집계·투영은 WBS 진행률에 자동 반영된다. 데이터 모델 상세는 `../data-model.md`.

## 7. 상태 및 전이

세부 업무 상태는 공통 상태 체계를 따른다.

```text
진행예정 → 진행중 → 완료
```

- 진행예정: 등록만 되고 아직 착수 전.
- 진행중: 작업이 진행되는 상태. WBS에서 초록색 막대로 표시되는 구간과 대응.
- 완료: 종료된 상태. WBS에서 파란색 막대로 표시.
- 역전이(완료 → 진행중 등)는 재작업 시 허용한다.
- 상태 값의 정식 정의와 색상 규칙은 `../status-taxonomy.md`를 단일 기준으로 삼는다.

## 8. 주요 액션 · 인터랙션

- 항목 추가: 구분·중분류·세부 업무를 새로 등록한다. 세부 업무 추가 시 상위 노드와 초기 상태를 지정한다.
- View: 구분 또는 세부 업무의 상세/요약을 확인한다.
- 상태 필터: 선택한 상태의 세부 업무만 남긴다. 필터 결과가 비면 상위 노드는 접히거나 비어 있음으로 표시한다.
- 펼침/접힘: 구분 노드 단위로 하위 항목을 열고 닫는다.
- 세부 업무 상태·일정 변경: 갱신하면 연결된 WBS 막대·진행률에 자동 반영된다.

## 9. 관련 사용자 흐름

업무 계획 관리 흐름의 시작점이 Task IA다.

```text
Task IA
→ 업무 항목 등록(구분/세부 업무)
→ 상태 지정(진행예정/진행중/완료)
→ WBS 일정 반영
→ 진행률 확인
```

- Task에서 정의한 세부 업무가 WBS의 일정표 행으로 연결되고, WBS에서 계산된 진행률이 다시 계획 대비 현황을 보여 준다.
- 전체 흐름 카탈로그는 `../ia.md`의 사용자 흐름 절을 참조.

## 10. 엣지 케이스 / 예외

- 등록된 업무 없음: 빈 상태 안내와 항목 추가 유도.
- 세부 업무가 없는 구분: 구분만 존재하고 하위가 비어 있는 경우, 상태 집계는 미지정으로 처리.
- 상태 필터 결과 없음: "해당 상태의 업무가 없습니다" 안내를 노출하고 트리 골격은 유지.
- 삭제/이동 시 하위 노드 처리: 구분·중분류 삭제는 확인 후 하위를 연쇄 삭제한다. 이동은 하위 세부 업무를 함께 옮긴다(재부모).
- 일정 미입력 세부 업무: 파생된 WBSItem에 일정이 없으면 그리드에 막대 없이 표시되고, 진행률은 상태 기반(완료=100·진행중=50·진행예정=0)으로 산출한다.
- 계층 깊이: 구분–중분류–세부 업무 3단을 넘어서는 입력은 허용하지 않는다.

## 12. 오픈 이슈

- 중분류(group)는 현재 화면에서 노출하지 않고 구분–세부 2단으로 운영한다(스키마에는 유지). — 결정 완료.
- 구분/중분류의 상태·진행률은 하위 세부 업무 집계로 자동 산출한다. — 결정 완료.
- 구분·중분류 삭제는 확인 후 연쇄 삭제, 이동은 재부모(하위 동반 이동). — 결정 완료.
- 세부 업무에 담당자·기한을 부여한다(Task가 태스크 정의처). — 결정 완료.
- Task 세부 업무 등록 시 WBSItem 1:1 자동 파생. — 결정 완료.
